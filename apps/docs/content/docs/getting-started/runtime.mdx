## Architecture

Scripts run inside the bot Lua runtime and use supported APIs exposed as global tables. Core helper libraries in `Scripts/core` wrap those bindings into a stable, validated public API.

## Cooperative execution

Script execution is **cooperative**, not preemptive. The runtime can only switch to other work when your script yields control. Blocking the runtime with a tight loop will freeze the bot.

### Yielding correctly

Use one of these patterns for repeating or delayed logic:

| Pattern | Use case |
| --- | --- |
| `Module.Every(name, fn, delayMs)` | Repeating task on an interval |
| `Module.After(name, fn, delayMs)` | One-shot delayed callback |
| `Module.New(name, fn, delayMs)` | Custom recurring module |
| `wait(ms)` | Yield inside a long-running coroutine |

HTTP and WebSocket operations also yield cooperatively while waiting. They must be started from a managed script coroutine (`init()` is managed and may call them).

**Avoid** busy loops:

```lua
-- BAD: freezes the runtime
while true do
    -- work without wait()
end
```

## Nil-safe design

Many APIs return `nil` when:

- The player is not logged in or not available
- A required capability is not loaded
- Game state does not exist yet (e.g. no target, closed container, no NPC trade window)

Always validate before use:

```lua
if not Self.IsAvailable() then
    return
end

local hp = Self.GetHealthPercentage()
if type(hp) ~= "number" then
    return
end
```

Capability wrappers such as `Inventory` and `NpcTradeStorage` can return nil/false when the corresponding game state is unavailable. Check `IsAvailable` / capability methods where provided.

Wrap risky blocks in `pcall` so one failure does not kill the entire script:

```lua
local ok, err = pcall(function()
    -- runtime logic
end)
if not ok then
    print("Error: " .. tostring(err))
end
```

## Module naming convention

Each module table is exposed in PascalCase. Some modules expose nested namespaces:

- `Cooldowns.Spell.IsReady(words)`
- `Cavebot.Walker.Resume()`
- `Engine.Healer.GetSpells()`
- `Engine.MagicShooter.GetEntries()`
- `Storage.Global.Get(key)`

Do not infer a `Query` or `Actions` namespace unless it is explicitly documented. Compatibility aliases are not part of the supported surface.

## Features API restriction

The Objects Dumper feature is reserved and intentionally blocked from public feature API paths. Do not attempt to enable, disable, or query it from scripts. Prefer `Engine.Features` for new scripts; the older global `Features` table remains for compatibility.

## Network constraints

- HTTP has no callback lifecycle API
- WebSockets use explicit `Receive`; there are no `onOpen`, `onClose`, `onError`, `onRedirect`, or automatic reconnect callbacks
- `validusbot.net`, its subdomains, and the registered service host are blocked
- A script may have at most four pending HTTP requests and four simultaneous WebSockets

## Hotkey constraints

- **alt** cannot be used with `Events.RegisterKeyEvent` / `Hotkeys.RegisterCombo`, but it is supported by `Hotkeys.SendCombo` / `Hotkeys.SendKey`
- Extended keys (insert, delete, home, end, page up/down, arrows) default to `extended = true` in `Hotkeys.ParseCombo`
- Standard Lua `table.concat` is allowed in sandboxed user scripts

See [Hotkeys API Reference](/docs/api-reference/hotkeys) for registration and send helpers.
